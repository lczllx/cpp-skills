## 1. LLM 底层原理

### 1.1 Token 与上下文窗口

- **Token 是什么？怎么计算的？** — Token 是文本处理的最小单位。英文：1 Token ≈ 4 个字符或 0.75 个单词；中文：1 Token ≈ 1-2 个汉字。GPT 系列使用 BPE（字节对编码）分词器，中文 Token 消耗通常比英文高
- **上下文窗口（Context Window）是什么？** — 模型一次能处理的最大 Token 数量，包括输入和输出。超过这个长度会被截断（Truncation）
- **上下文窗口满了怎么办？** — 方案：滑动窗口（Sliding Window）截断历史；摘要脱水（Summarization）；分层记忆（Hierarchical Memory）；RAG 检索相关片段替代完整历史
- **为什么中文比英文消耗更多 Token？** — BPE 分词器对不同语言效率不同，汉字在训练语料中复现率低，导致分词粒度更细；中文无空格分隔，需要更多 Token 表示相同语义

### 1.2 Temperature 与采样参数

- **Temperature 是什么？如何影响输出？** — Temperature 控制输出的随机性。Temperature=0 时，几乎总是选择最高概率的词（Deterministic）；Temperature=1 时，概率分布保持原始状态；Temperature>1 时，更容易选择低概率词（更随机、更创造性）
- **什么场景用什么 Temperature？** — 0：代码生成、结构化输出、翻译（确定性）；0.3-0.7：对话、总结（平衡）；1.0+：创意写作、头脑风暴（多样性）
- **Top-p 采样和 Top-k 采样的区别？** — Top-k：只从概率最高的 k 个词中采样；Top-p：从累计概率达到 p 的最小词集中采样。Top-p 更常用，能动态调整候选词数量
- **结构化输出（JSON Mode）为什么还是要指定 Temperature？** — 即使指定了 JSON 格式，模型仍需要在多个合法选项中做选择，Temperature 控制这种选择的随机性；Temperature=0 可提高格式稳定性但不能完全保证

### 1.3 模型能力与局限性

- **幻觉（Hallucination）是什么？为什么会产生？** — 幻觉指模型生成看似合理但实际错误或不存在的内容。根源：LLM 是语言模型而非知识数据库；训练数据存在错误或偏见；缺乏实时验证机制；注意力机制会在不存在的模式中"找到"联系
- **如何缓解幻觉？** — RAG（检索增强生成）；提示词约束（要求引用原文）；CoT（链式推理）让推理过程透明；工具调用验证外部知识；事实核查（Fact Checking）层
- **LLM 的"失忆"问题是什么？** — 上下文长度限制导致长对话中早期信息被截断；位置编码衰减导致模型对上下文中部信息关注度最低（Lost in the Middle）；固定上下文窗口外的长期记忆需要外部存储
- **长上下文模型（如 100K+ Token）有什么新问题？** — 计算成本二次增长；注意力分散（难以在超长上下文中准确定位关键信息）；依然存在截断风险；推理延迟高

## 2. AI Agent

### 2.1 Agent 核心概念

- **AI Agent 是什么？核心公式？** — Agent = LLM + Planning + Memory + Tools。能够感知环境、进行决策并执行动作的自主软件系统，以 LLM 为大脑完成复杂任务
- **Agent Loop 是什么？工作流程？** — 本质是一个 while 循环：LLM 推理 → 工具调用 → 上下文更新 → 重复，直至任务终止。关键组件：System Prompt、可用工具列表、历史上下文、终止条件
- **Agent vs Workflow vs 传统编程的本质区别？** — 传统编程和 Workflow 是人在做决策，Agent 是 AI 在做决策。关键区别在于灵活性、门槛和维护成本
- **Agent 当前的核心挑战？** — 上下文窗口限制；幻觉问题；Token 经济性（长任务成本高）；工具安全边界（Prompt Injection）；规划能力上限；可观测性不足

### 2.2 ReAct 与推理范式

- **ReAct 是什么？核心思想？** — ReAct（Reasoning + Acting）将思维链推理与外部环境交互结合，让 AI "走一步看一步"，通过"思考→行动→观察"的循环完成任务，显著减少幻觉、提升可解释性
- **ReAct 和 Plan-and-Execute 的区别？** — ReAct：动态逐步规划，每步基于上步观察调整；Plan-and-Execute：先制定全局计划，再按步骤执行。ReAct 更灵活但上下文持续增长；Plan-and-Execute 更可控但缺乏动态调整能力
- **Reflection 反思模式是什么？** — 赋予 Agent 自我纠错能力，通过自然语言反馈强化模型行为。包括 Reflexion（失败后反思）、Self-Refine（输出后批评改进）、CRITIC（外部工具验证）
- **Multi-Agent 系统是什么？** — 多个独立 Agent 协作完成复杂任务。主流架构：Orchestrator-Subagent（编排者-子代理）；Peer-to-Peer（对等协作）。优点：并行效率高、专业分工；缺点：通信开销大，调试复杂

### 2.3 Context Engineering

- **Context Engineering 是什么？** — 为 LLM 构建高信噪比信息输入环境的技术。核心目标是在有限 Token 窗口内以最低信噪比为模型提供最精准的决策依据
- **System Prompt 怎么设计最有效？** — 高度结构化的 Markdown 格式：角色设定、核心目标、严格约束、标准执行流、输出格式。使用分隔符区分不可信外部数据
- **短期记忆 vs 长期记忆？** — 短期记忆：当前会话上下文，通过滑动窗口管理；长期记忆：外部知识库（向量数据库或知识图谱），通过检索注入
- **上下文太长怎么优化？** — 低优先级（可折叠）：早期对话历史压缩为摘要；中优先级（可精简）：RAG 检索资料二次裁切；高优先级（绝对保护）：系统约束和核心工具描述不可丢失；Context Caching 降低并发成本

### 2.4 多智能体系统

- **A2A 协议是什么？解决什么问题？** — Agent-to-Agent 通信协议，专为 AI 智能体间高效协作设计。使用结构化 JSON（TaskID、Dependencies、AcceptanceCriteria）替代自然语言，降低 Token 消耗并避免格式解析错误
- **Agentic Workflows 是什么？** — 吴恩达倡导的宏观概念，核心四大设计模式：Reflection（反思）、Tool Use（工具使用）、Planning（规划）、Multi-agent Collaboration（多智能体协作）
- **MCP 和 A2A 的关系？** — MCP 是 AI 与外部工具的通信协议（相当于微服务的 REST/gRPC）；A2A 是 Agent 与 Agent 之间的通信协议（相当于微服务间的消息队列）

## 3. RAG 检索增强生成

### 3.1 RAG 核心原理

- **RAG 是什么？解决什么问题？** — Retrieval-Augmented Generation，检索增强生成。解决 LLM 知识过时、幻觉、知识截止日期等问题，通过检索外部知识库为 LLM 提供最新、最准确的上下文
- **RAG 的完整流程？** — 索引阶段：文档加载 → 文本分割（Chunking）→ 向量化 → 存入向量数据库；查询阶段：用户问题向量化 → Top-K 检索 → 上下文组装 → LLM 生成
- **RAG vs Fine-tuning 的区别？各适合什么场景？** — RAG：动态知识、频繁更新、需要可解释性、降低成本；Fine-tuning：固定模式、风格学习、实时性要求低、复杂推理调整。两者可结合使用
- **RAG 的核心局限性？** — 检索质量依赖 Chunking 策略和向量模型效果；上下文长度限制导致无法引入所有相关文档；多跳推理困难；跨语言检索效果差

### 3.2 文档分块策略

- **Chunk Size 怎么选？** — 固定分块：简单但效果一般（通常 512-1024 Token）；句段落分块：尊重语义完整性；递归字符分块：按层级分割（段落→句子→词）；语义分块：基于 Embedding 自动判断分割点
- **Chunk Size 太大或太小各有什么问题？** — 太大：引入过多无关上下文，稀释关键信息，增加 Token 成本和推理延迟；太小：丢失跨 Chunk 语义，上下文碎片化，召回片段不完整
- **重叠分块（Overlap）的作用？** — 在相邻 Chunk 之间保留重叠内容，防止语义在分割处断裂。常用 overlap=10-20%，但增加 Token 消耗
- **如何根据不同文档类型调整分块策略？** — 结构化文档（PDF/HTML）：先识别标题层级结构，按层级分块；代码：按函数/类分块，保留缩进和上下文；对话记录：按轮次分块

### 3.3 向量检索与索引

- **常见向量索引算法？** — FLAT（暴力检索，准确但慢）、IVF（倒排索引，聚类加速）、HNSW（分层导航小世界图，高召回低延迟）、PQ（乘积量化，压缩存储）
- **HNSW 的原理？优缺点？** — 分层图结构，从上到下逐层搜索。上层跳步远，下层精细。优点：召回率高、延迟低；缺点：内存占用大、构建慢、不支持动态更新
- **余弦相似度 vs 点积 vs 欧氏距离？** — 余弦相似度：衡量方向，不受向量长度影响，适合文本 Embedding；点积：综合考虑方向和幅度，适合已归一化的向量；欧氏距离：衡量绝对距离，适合图像等密集特征
- **向量数据库怎么选？** — Milvus（大规模、高性能、支持混合检索）、Pinecone（托管方便、Serverless）、Chroma（轻量、本地开发）、Qdrant（支持过滤、性能好）、Weaviate（原生支持混合搜索）

### 3.4 召回质量优化

- **召回率低怎么排查和优化？** — 检查向量模型是否与文档领域匹配；调整 Chunk Size；增加 Overlap；优化检索策略（混合检索：向量+关键词）；使用重排序（Re-ranker）
- **Re-ranker 的作用？** — 两阶段检索：粗召回（向量检索，Top-K 大）→ 精排（Cross-Encoder 重排序，输出 Top-N 小）。Cross-Encoder 能考虑更精细的语义匹配，弥补向量检索的精度损失
- **混合检索是什么？** — 结合向量检索（语义相似）和关键词检索（BM25/TF-IDF）。适合同时需要语义理解和精确关键词匹配的场景。常用方案：Reciprocal Rank Fusion（RRF）融合多路结果
- **如何处理跨语言检索？** — 使用多语言 Embedding 模型（如 sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2）；翻译查询或文档；使用 CLUE（跨语言理解）专用模型
- **如何判断 RAG 系统效果差是检索问题还是生成问题？** — 单独测试检索模块：用 Ground Truth 答案验证召回内容是否包含；看生成回答是否只依赖召回内容（如果没用到召回片段说明检索没问题）；用完美检索结果输入生成器测试

### 3.5 RAG 优化验证与度量标准（重要追问链）

> 当候选人提到 RAG 优化（chunk 重排、重写、混合检索等）时，必须连续追问验证，避免"堆砌技术"而不知效果。

**追问链（必须连续追问）：**

1. **第一层：你怎么验证的？**
   - 期望回答：有人工评估数据集、与 baseline 对比等
   - 如果只说"体感提升了"→ 继续追问

2. **第二层：量化指标是什么？提升了多少%？**
   - 期望回答：Recall@K、MRR、RAGAS、Faithfulness 等指标
   - 如果无法给出具体数字 → 继续追问

3. **第三层：对比实验怎么做？消融实验有吗？**
   - 期望回答：逐步去掉每个优化项，看效果变化
   - 例如：去掉 Re-ranker 效果降了多少？去掉混合检索呢？

4. **第四层：知识库规模多大？多少条数据？**
   - 期望回答：应有一定规模（建议 500+ 文档）才能说明优化有效
   - 如果只有 10-20 条数据 → 指出这个规模本身就不存在检索困难

**常见错误回答模式（面试官需质疑）：**
- ❌ "体感上提升了" → 体感不是度量标准
- ❌ "用了社区常用的方法" → 常用不代表有效
- ❌ "知识库有 14 篇文章" → 数据量太小，优化无意义
- ❌ "加了 Re-ranker 效果好了" → 消融实验有吗？具体提升多少？

**正确思路（引导候选人思考）：**
- ✅ 先建度量标准，再做优化
- ✅ 每个优化项单独验证，避免堆砌
- ✅ 知识库规模要足够大（500+）才能说明问题
- ✅ 用离线指标 + 线上用户反馈双重验证

## 4. MCP 协议与工具调用

### 4.1 Function Calling

- **Function Calling 是什么？底层原理？** — 允许 LLM 调用外部工具/函数的能力。模型输出结构化的函数调用请求（而非自然语言），应用解析并执行后返回结果。底层依赖 JSON Schema 描述函数签名
- **好的工具描述（Function Description）怎么写？** — 必须明确说明"何时该调用"和"何时不该调用"；参数描述包含格式要求和典型示例值；描述质量直接决定 Agent 的决策准确性
- **Function Calling 的局限性？** — 依赖模型是否支持该能力；参数填充仍可能出错；多函数调用时顺序依赖难以处理；实时性要求高的场景延迟高
- **Function Calling vs Agent Skills 的区别？** — Function Calling：调用外部 API/代码，模型只看到 JSON Schema 签名；Agent Skills：以自然语言指令集为主，模型能看到完整执行逻辑（SKILL.md），更灵活

### 4.2 MCP 协议

- **MCP 是什么？为什么被称为"AI 领域的 USB-C"？** — Model Context Protocol，Anthropic 2024 年 11 月推出的开放协议。统一了 AI 应用与外部工具的通信规范，实现一次接入、全局复用，被比喻为 USB-C 接口
- **MCP 的三大核心原语？** — Tools（可执行函数，供 LLM 主动调用）、Resources（只读数据，供 Agent 按需读取）、Prompts（可复用提示词模板）
- **MCP Server 怎么开发？** — 定义工具 JSON Schema → 实现工具执行逻辑 → 通过 MCP SDK 暴露为 Server → 宿主程序连接后自动发现工具。主流语言均有 SDK 支持
- **MCP 和 Function Calling 的关系？** — JSON Schema 是底层数据格式标准（定义模型如何"读懂"工具）；MCP 是在其之上的通信协议层（定义工具如何"标准化接入"宿主程序）。两者正交，可叠加使用

### 4.3 工具注册与安全

- **Prompt Injection 是什么？怎么防护？** — 攻击者通过构造外部输入试图覆盖或篡改 Agent 系统指令。防护：执行层（沙箱隔离、权限最小化）；认知层（Prompt 隔离、分隔符）；决策层（高危操作人机审批）
- **工具注册的最佳实践？** — 避免工具幻觉：描述必须准确、无歧义；动态注册：按需加载而非一次性暴露所有工具；版本管理：工具签名变更需同步更新描述
- **如何评估工具调用质量？** — 工具选择准确率（是否调用了正确的工具）；参数填充正确率（参数值是否正确）；调用成功率（执行是否成功）；端到端任务完成率

## 5. AI 编程实战

### 5.1 AI 编程工具使用

- **你用过哪些 AI 编程工具？** — 考察候选人对 AI 编程工具的实际使用经验。主流工具：GitHub Copilot、Cursor、Windsurf、Trae、Claude Code 等
- **AI 编程工具的核心原理？** — 上下文感知的代码补全（基于当前文件+相邻文件+注释）；自然语言转代码；代码审查与优化建议；多模态理解（截图、UI）
- **如何写出好的 AI 编程 Prompt？** — 明确任务而非模糊描述；提供上下文（文件路径、相关代码、报错信息）；指定输出格式；分步骤引导而非一步到位；使用代码模板减少幻觉
- **AI 编程的局限性？** — 对复杂业务逻辑理解不足；容易产生看似正确但有 bug 的代码；缺乏对运行时行为的理解；长对话中上下文管理困难

### 5.2 AI 辅助编程最佳实践

- **什么任务适合 AI 编程工具？** — 样板代码生成、API 调用示例、代码重构、Bug 修复、测试用例生成、文档生成、代码解释
- **什么任务不适合 AI 编程？** — 复杂业务逻辑设计、系统架构决策、性能优化（需要实测）、安全性敏感代码、新技术/框架的第一次使用
- **如何使用 AI 做代码审查？** — 提供完整上下文（PR 链接或代码 diff）；明确审查重点（性能、安全、可读性）；AI 擅长发现浅层问题；复杂业务逻辑仍需人工判断
- **AI 编程的未来趋势？** — 更强的多模态能力（直接理解 UI 设计稿）；Agent 化（AI 自动完成从需求到代码到测试的完整流程）；原生代码生成（而非基于现有代码续写）

### 5.3 AI Coding Agent 架构（Claude Code / Cursor）

> 面试高频：面试官可能直接问"你了解 Claude Code / Cursor 的源码吗？有特别了解的模块吗？"

- **AI Coding Agent 的核心架构是什么？** — 本质是一个 Agent Loop（while 循环）：读取用户输入 → 组装上下文（System Prompt + 对话历史 + 工具定义）→ 调用 LLM → 解析响应 → 如果包含工具调用则执行工具并将结果追加到上下文 → 循环，直到模型输出纯文本（不再调用工具）为止。Claude Code、Cursor Agent、Copilot Workspace 底层都是这个模式
- **Agent Loop 的关键组件？** — (1) System Prompt：定义角色、约束、可用工具列表；(2) 对话历史（Message History）：user/assistant/tool 三种角色的消息序列；(3) 工具注册表（Tool Registry）：通过 JSON Schema 描述工具签名，注入 system prompt 或 tools 参数；(4) 工具执行器（Tool Executor）：解析模型输出的工具调用 → 路由到对应实现 → 执行 → 返回结果；(5) 终止条件：模型不再发起工具调用 / 达到最大轮次 / 用户中断
- **Claude Code 的工具有哪些？怎么工作的？** — 核心工具包括：Read（读取文件）、Edit（编辑文件）、Write（写入文件）、Bash（执行命令）、Glob（文件搜索）、Grep（内容搜索）、Agent（子代理）等。每个工具通过 JSON Schema 定义参数，模型在推理时决定调用哪个工具、传什么参数。工具执行结果作为 tool role 消息追加到对话历史，模型在下一轮推理时看到结果并决定下一步
- **上下文管理怎么做的？** — 对话越长，上下文 Token 越多。核心策略：(1) 上下文压缩/截断（Truncation）：当接近上下文窗口上限时，压缩早期对话为摘要；(2) 按优先级保护：System Prompt 和最近几轮对话始终保留，中间的历史消息优先压缩；(3) 工具结果裁剪：大文件读取结果可能被截断，只保留关键部分
- **权限与安全模型怎么设计的？** — 工具调用分为自动允许和需要用户确认两类。读取类操作（Read、Glob、Grep）通常自动允许；写入类操作（Edit、Write、Bash）需要用户确认或在特定权限模式下自动允许。防止 Prompt Injection：工具返回的外部数据标记为不可信，避免覆盖系统指令
- **子代理（Sub-agent）机制是什么？** — Claude Code 的 Agent 工具可以启动一个独立的子代理，拥有独立的上下文窗口。用途：(1) 隔离大量搜索结果，避免污染主上下文；(2) 并行执行独立任务；(3) 将复杂研究任务委托给子代理，只取回结论。子代理完成后返回结果给主代理，主代理基于结果继续推理

### 5.4 Skill 机制与 AI 工作流编排

> 面试高频：面试官可能问"Skill 在 Claude Code 里是怎么运作的？""你的 Skill 工作流是怎么串联起来的？"

- **Skill 是什么？和 MCP / Function Calling 有什么区别？** — Skill 是以自然语言（Markdown）编写的可复用能力模块。与 Function Calling 的区别：Function Calling 通过 JSON Schema 定义参数、由代码执行；Skill 通过 SKILL.md 文件定义完整的执行指令，由 LLM 阅读并按指令行动。Skill 更灵活，适合复杂流程编排；Function Calling 更精确，适合单一工具调用
- **Skill 在 Claude Code 中的加载机制？** — 两阶段加载：(1) 轻量注入：每个 Skill 的 name + description（几十个 Token）始终注入 system prompt，LLM 可以"看到"所有可用 Skill 的摘要；(2) 按需激活：当用户请求匹配某个 Skill 的 description 时，LLM 决定激活该 Skill，此时完整的 SKILL.md 内容被加载到上下文中。这种设计平衡了上下文消耗和能力覆盖
- **Skill 的 description 为什么很重要？** — description 是 LLM 判断"是否需要激活这个 Skill"的唯一依据。写得太模糊 → 该激活时不激活；写得太宽泛 → 不该激活时误触发。好的 description 应该包含：触发条件（TRIGGER when）、不触发条件（DO NOT TRIGGER when）、核心能力描述
- **如何设计多 Skill 协作的工作流？** — 主 Skill + 子 Skill 模式：主 Skill（SKILL.md）定义整体流程步骤（第一步做什么 → 第二步做什么 → 异常处理），子 Skill 负责独立能力（获取 Cookie、Git 操作、代码分析等）。主 Skill 通过自然语言指令告诉 LLM 何时调用哪个子 Skill。类比软件工程中的编排模式（Orchestrator Pattern）
- **Skill vs MCP Server 各适合什么场景？** — Skill 适合：复杂流程编排、需要 LLM 判断和决策的任务、需要灵活调整的工作流；MCP Server 适合：确定性的工具调用、需要精确参数传递的 API 封装、需要跨多个 AI 客户端复用的标准化能力。两者可组合使用：Skill 编排流程，流程中调用 MCP Server 提供的工具

### 5.5 Prompt Cache 与性能优化

> 面试高频：了解 AI 应用的性能优化和成本控制

- **Prompt Cache 是什么？原理？** — Anthropic 的 Prompt Cache 机制：当多次 API 调用的 prompt 前缀相同时，后续调用可以复用前缀的计算结果（KV Cache），减少首 Token 延迟（TTFT）和计算成本。缓存按 prompt 前缀严格匹配——只要前缀相同，后面追加的内容不影响缓存命中
- **Prompt Cache 的 TTL 是多少？有什么影响？** — 缓存有 5 分钟 TTL（Time To Live）。超过 5 分钟未使用则失效，下次调用需要重新计算。对于 AI Coding Agent：Agent Loop 中连续的工具调用间隔通常在秒级，缓存命中率高；但如果用户长时间不操作，缓存会过期
- **如何设计 prompt 结构来最大化缓存命中？** — 把稳定内容放前面、变化内容放后面。System Prompt（稳定）→ 工具定义（稳定）→ 早期对话历史（相对稳定）→ 最近对话（变化）。这样前缀尽可能长且稳定，缓存命中率最高
- **Prompt Cache 对成本的影响？** — 缓存命中的 Token 只收 1/10 的价格（以 Claude 为例），显著降低高频调用场景的成本。但缓存未命中时成本不变。对于 Agent 场景，一个任务可能有几十轮工具调用，缓存优化的效果非常明显

## 6. 前端 AI 落地实战场景

> 本节覆盖前端工程师在实际项目中落地 AI 能力的常见场景、方案选型和工程化策略。面试中"你做过哪些 AI 相关的落地"是高频问题，候选人需要能讲清楚方案演进路径和工程化细节。

### 6.1 LLM 结构化输出与前端渲染

- **LLM 输出如何转化为前端可渲染的结构？** — 常见方案：(1) 直接生成 JSON：让模型输出符合前端组件 props 格式的 JSON，前端解析后渲染。优点是直接可用；缺点是格式不稳定，模型可能输出非法 JSON；(2) 中间格式转换：让模型输出标准化中间格式（如 Mermaid、Markdown、SVG），再通过解析器转为前端元素。优点是格式更可控；缺点是多一层转换开销；(3) 流式渲染：通过 SSE / ReadableStream 逐 Token 接收并实时渲染，需要处理不完整 JSON/Markdown 的中间状态
- **典型场景：自然语言生成流程图/图表** — 方案演进路径：直接生成目标 JSON（不稳定）→ 引入中间层 DSL 如 Mermaid 语法（更可控）→ Prompt 工程给示例提升质量 → 脚本清洗修复格式问题 → 降级兜底（生成失败则输出图片）。每一层都是解决上一层暴露的问题
- **Prompt 工程在结构化输出中的作用？** — 给模型提供 few-shot 示例（输入-输出对）；用 JSON Schema 约束输出格式；指定严格的格式规范（如"只输出合法 Mermaid 语法，不要添加任何解释文字"）；设置 Temperature=0 提高格式稳定性
- **流式输出场景下前端怎么处理？** — SSE（Server-Sent Events）或 fetch + ReadableStream 接收逐 Token 输出；前端维护一个 buffer 拼接文本并实时渲染；对于 Markdown 内容需要处理不完整语法（如未闭合的代码块）；React 中通常用 useRef 存 buffer + useState 触发渲染，避免高频 setState 导致的性能问题

### 6.2 多模态在前端场景的应用

- **前端场景中多模态能力怎么用？** — (1) 画布/白板场景：将画布元素截图 + 结构化数据一起发给多模态模型，让模型理解空间关系；(2) UI 审查：截图 + 设计稿对比，让模型找出还原度问题；(3) 文档理解：上传文档截图，让模型提取结构化信息（表格、图表、手写内容）；(4) 辅助开发：截图描述 UI → 生成对应代码
- **为什么纯文本坐标不够，需要截图？** — XY 坐标只能表示元素的绝对位置，无法表达视觉上的空间关系（哪些元素在同一行、哪些是嵌套关系、哪些是连线关系）。多模态模型通过"看图"可以直接理解人类视觉感知到的布局关系，准确率显著高于纯坐标解析
- **多模态输入的成本和优化？** — 图片 Token 消耗大（一张高分辨率截图可能消耗数千 Token）。优化策略：降低截图分辨率（在保证可识别的前提下）；裁剪只保留相关区域而非全屏截图；对于重复请求，缓存图片分析结果；权衡：截图质量越高准确率越好，但成本和延迟也越高

### 6.3 AI 功能的工程化兜底策略

- **AI 输出不确定性怎么处理？** — AI 功能的核心工程化挑战是**输出不可靠**。必须在架构层面设计兜底策略：(1) 格式校验层：验证输出是否符合预期格式（JSON Schema 校验、正则匹配）；(2) 清洗修复层：对常见格式问题做自动修复（补全缺失的括号、移除非法字符）；(3) 降级兜底层：修复失败时降级为安全输出（如图片、纯文本、默认模板）；(4) 重试机制：可选的自动重试（注意设置最大重试次数和超时）
- **如何监控 AI 功能的线上质量？** — 埋点维度：AI 功能调用次数、成功率、格式校验通过率、降级触发率、用户手动修改率（反映生成质量）、平均响应延迟。告警阈值：格式校验失败率 > 10% 或降级触发率 > 20% 时告警。长期通过数据驱动优化 Prompt 和兜底策略
- **AI 功能的用户体验设计？** — 加载态：流式输出场景展示逐字渲染动画（而非空白等待）；错误态：降级后给用户明确提示（"生成结果可能不完美，你可以手动调整"）；可编辑性：AI 生成的结果必须支持用户二次编辑；反馈机制：提供"结果不满意"入口收集负面反馈，用于后续优化

## 7. AI 系统设计高频场景

### 7.1 客服 Agent 设计

- **整体架构？** — 用户请求 → 意图识别（Intent Detection）→ 路由（Routing）→ 子 Agent/技能处理 → 回复生成。关键组件：对话管理、意图识别、知识库检索、情感识别
- **意图识别怎么做？** — 方案：关键词匹配（简单场景）、分类模型（固定意图集）、LLM 零样本分类（开放意图）、混合方案（先用规则再用模型）
- **如何处理无法回答的问题？** — 转人工策略（置信度低于阈值时触发）；多轮澄清（反问获取更多信息）；FAQ 兜底回复；升级机制（普通客服→专家）
- **对话状态管理？** — 短期状态（当前会话）：槽位填充（Slot Filling）；长期状态（跨会话）：用户画像、偏好记录、历史问题。技术选型：内存（Redis）/数据库

### 7.2 知识库问答系统

- **如何设计一个企业知识库问答系统？** — 数据层：多源文档接入（Notion/Confluence/API）→ 文档解析 → Chunking → 向量化 → 索引；检索层：query 解析 → 混合检索 → 重排序 → 上下文组装；应用层：LLM 生成 → 答案校验 → 引用溯源
- **多跳推理怎么做？** — 迭代检索（每次检索一个事实，逐步构建答案）；链式推理（Chain-of-Thought 让模型显式推理）；图结构（构建文档关系图，用 GNN 或路径推理）
- **如何处理表格、图表等非文本内容？** — 表格：转为 Markdown 或 JSON 结构化表示；图表：Alt 文本描述提取或视觉模型描述；PDF：布局分析识别标题/正文/图表区域
- **如何评估 RAG 系统效果？** — 检索指标：Recall@K、MRR、NDCG；生成指标：RAGAS（综合评估）、faithfulness（是否依赖召回内容）、answer relevancy；人工评估：准确性、完整性、流畅性

### 7.3 具身智能与多模态

- **什么是具身智能（Embodied AI）？** — AI 能理解物理世界并通过身体（机器人、AR/VR 设备）与环境交互。结合计算机视觉、自然语言理解、机器人控制的综合能力
- **Computer Use 是什么？** — 让 Agent 能通过屏幕、鼠标、键盘与图形界面交互。Agent 能"看见"屏幕截图，理解 UI 元素，并执行点击、输入等操作
- **多模态 Agent 的应用场景？** — UI 自动化测试；文档理解（发票、合同）；视频分析（监控、内容审核）；具身机器人；辅助驾驶/自动驾驶
- **多模态 Agent 的挑战？** — 视觉理解延迟（高分辨率截图 Token 消耗大）；动作执行的精确性；跨模态对齐（视觉动作与语言指令对应）；实时性要求（机器人控制需要低延迟）
